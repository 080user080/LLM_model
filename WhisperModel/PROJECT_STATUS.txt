
# Проект: Асистент МАРК (Voice/Text Assistant with Agent Capabilities)

## 1. Загальний опис

**МАРК** — це персональний голосовий та текстовий асистент для Windows, написаний на Python.  
Головна мета — створити автономного агента, який отримує завдання українською мовою, самостійно планує кроки, виконує їх (наприклад, пише код, відкриває програми, працює з файлами) та надає готовий результат.

Особливості:
- Працює **повністю локально** (STT, TTS, LLM через LM Studio), не потребує інтернету.
- Має графічний інтерфейс (GUI) для текстового спілкування, підтвердження дій та перегляду історії.
- Підтримує голосове введення команд (наразі у режимі тестування, планується використовувати лише для диктування тексту в поле вводу).

## 2. Архітектура (коротко)

```
[GUI (Tkinter)] <--черга повідомлень--> [AssistantCore] --> [VoiceAssistant] --> [FunctionRegistry]
                                                                         │
                                                              ┌──────────┴──────────┐
                                                              ▼                     ▼
                                                        [CORE модулі]          [НАВИЧКИ]
                                                        - кеш                 - виконання Python
                                                        - диспетчер           - дебаг коду
                                                        - пісочниця безпеки   - відкриття програм
                                                        - стрімінг            - створення/редагування файлів
                                                              │
                                                              ▼
                                                         [LM Studio]
```

- **AssistantCore** — ініціалізує всі сервіси (STT, TTS, слухач, реєстр функцій).
- **VoiceAssistant** — обробляє команди (кеш → швидкі маршрути → LLM).
- **FunctionRegistry** — автоматично завантажує всі файли `aaa_*.py` з папки `functions` і робить їх доступними для виклику через LLM.
- **GUI** — окремий потік, безпечно взаємодіє з ядром через чергу повідомлень.

## 3. Поточний стан проекту (березень 2026)

### ✅ Що вже реалізовано та стабільно працює

- **Ядро та GUI**  
  - Повноцінний графічний інтерфейс на Tkinter (чат, поле вводу, діалоги підтвердження).  
  - Копіювання/вставка тексту в чаті та полі вводу працює коректно.  
  - Можливість надсилати команди текстом через GUI (основний режим взаємодії).

- **Розпізнавання мови (STT)**  
  - Whisper (large-v3) та українська модель w2v-bert-uk.  
  - Паралельний режим порівняння результатів для підвищення точності.  
  - **Примітка:** голосове розпізнавання нестабільне, тому **основною** взаємодією залишається текстовий ввід через GUI. Голос планується використовувати лише для диктування тексту в поле вводу (функція `voice_input`).

- **Синтез мовлення (TTS)**  
  - Українська модель StyleTTS2 (мультиспікер).  
  - Кешування згенерованих аудіо для швидкого повторного відтворення.  
  - Автоматичне призупинення прослуховування мікрофона на час озвучення відповіді.

- **Безперервне прослуховування (Continuous Listener)**  
  - Детекція початку та кінця фрази за рівнем гучності та паузами.  
  - Індикатор активності в консолі.  
  - **Планується відключити** для командного режиму, залишивши лише для функції голосового вводу тексту.

- **Обробка команд**  
  - Трирівнева система: кеш → диспетчер (regex) → LLM.  
  - Кешування пар «команда → відповідь» у JSON, з автоматичним виконанням відомої дії.  
  - Інтеграція з локальним LLM через LM Studio (зараз використовується звичайний запит, але модуль **стрімінгу** готовий — він дозволяє отримувати відповідь по словах, щоб користувач бачив процес «думання» в реальному часі).

- **Безпека та пісочниці**  
  - **SafetySandbox** — відкриття/закриття програм лише з whitelist, можливість підтвердження через GUI.  
  - **PythonSandbox** — ізольоване виконання Python‑коду в окремому процесі (таймаут 30с, заборона небезпечних викликів).

- **Навички (функції)**  
  - `execute_python` / `debug_python_code` — виконання та автоматичне виправлення Python‑коду.  
  - `open_program` / `close_program` — керування програмами.  
  - `create_file` / `edit_file` — робота з текстовими файлами.  
  - `voice_input` — запис голосу та вставка розпізнаного тексту в активне поле.  
  - `architect` — динамічне створення нових навичок за допомогою LLM.  
  - `confirm_action` — діалог підтвердження дій у GUI.

### ⚠️ Відомі обмеження / недоліки

- **Голосове керування нестабільне** — прийнято рішення використовувати його лише як допоміжний інструмент для введення тексту, а не для прямих команд.
- **Відсутній планувальник** — асистент виконує лише одну дію за раз. Для складних завдань потрібен модуль, який розбиває задачу на кроки.
- **Контекстна пам’ять обмежена** — зберігається лише останні 10 повідомлень у діалозі. Для агента потрібна пам’ять про проміжні результати, файли, змінні.
- **Дублювання функцій** — є два файли для відкриття програм (`aaa_open_program.py` та `aaa_programs.py`). Потребує об’єднання для зменшення коду (некритично).
- **Відсутнє автоматичне тестування результату** — після виконання дії агент не перевіряє, чи дійсно досягнуто бажаного (наприклад, чи створився файл). Потрібно в майбутньому.

## 4. План розвитку (Roadmap)

Проект рухається до стану **автономного агента**, здатного самостійно планувати та виконувати завдання.

### Етап 1. Стабілізація та рефакторинг (поточний)

- ✅ Зроблено: перехід на текстовий ввід через GUI як основний.
- 🔜 Об’єднати дублюючі функції для відкриття програм.
- 🔜 Відключити безперервне прослуховування для команд, залишивши його лише для `voice_input`.
- 🔜 Увімкнути стрімінг відповідей від LLM у GUI (показувати процес генерації).

### Етап 2. Впровадження планувальника (Planner)

- Створити модуль `core_planner.py`, який:
  - Отримує складне завдання від користувача.
  - За допомогою LLM генерує план дій (послідовність викликів функцій).
  - Відображає план у GUI для затвердження.
- Реалізувати менеджер виконання плану (послідовний запуск кроків, обробка помилок).

### Етап 3. Розширення контекстної пам’яті

- Додати сховище для проміжних результатів (змінні, файли, вивід команд).
- Навчити агента використовувати цю пам’ять у наступних кроках (наприклад, «візьми результат попереднього скрипта і відправ його на email»).
- Збільшити ліміт історії діалогу.

### Етап 4. Покращення безпеки та тестування

- Інтегрувати `confirm_action` у всі потенційно небезпечні операції.
- Додати можливість автоматичної перевірки результату після виконання дії (assert-стиль).
- Розширити набір навичок: HTTP‑запити, робота з архівами, запуск зовнішніх скриптів.

### Етап 5. Полірування та зручність використання

- Налаштування через GUI (вибір моделі, голосу, гучності).
- Збереження налаштувань у конфігураційному файлі.
- Можливість зберігати та завантажувати плани завдань.

## 5. Технічні примітки

- **Стрімінг відповідей LLM** — це спосіб отримувати відповідь частинами, щойно LLM їх генерує. Замість того, щоб чекати всю відповідь 10 секунд, користувач одразу бачить, як з’являються перші слова. Модуль `core_streaming.py` уже написаний, залишилося підключити його до GUI.

- **Залежності** — проект налаштований на конкретний ПК (шляхи до CUDA, ID мікрофона). Для перенесення на іншу машину потрібно буде оновити `config.py` та перевірити сумісність драйверів.

- **Мова програмування** — Python 3.10+. Використовуються бібліотеки: `torch`, `transformers`, `whisper`, `sounddevice`, `tkinter`, `pyautogui`, `requests`.
